Materials Commons Python API

API Overview

We assume that the reader has used (or is otherwise familiar with) the Materials Commons web site (https://materialscommons.org/). This site provides a User Interface to the Materials Commons server.

In this document, we describe the Python API to the Materials Commons. It is a Python module embodying the API  (Application Programmer Interface) that to the Materials Commons server. It supports creating, modifying, and deleting Materials Commons Projects and their parts: Experiments, Processes, Samples, Directories, and Files. These "objects of discourse" are depicted in this object-relation sketch.

The object with the white backgrounds (Project, Experiment, Process, File, Directory, and Dataset) are the main objects that one uses to create a project in Materials Commons and the Python API lets a user create and manipulate each of these object type, except for Dataset. Since Dataset's are used as the means of publishing data on the public data site for Materials Commons, we felt it best that the web site should be used for that final publication.

In the diagram above the lines between the objects a relationship, but do not specify the nature of that relationship. In this document, those relations can be thought of as "contains" in the following way:

  • a Project contains
    • (and consists of) a list of Experiments
    • a (single) root Directory
    • a list of files
    • a list of Samples
    • a list of Processes
  • an Experiment contains
    • a list of Files
    • a list of Samples
    • a list of Processes
  • a Directory contains
    • a list of Directories and Files

Access to the API

All API access is through function or method calls in the API (see the API Details section below). The API requires a configuration with a base url and an apikey.  The configuration is set up using the file ~/.materialscommons/config.json (where ~ denoted the users home directory) .
The URL for the API support by the production version of the Materials Commons is: materialscommons.org/api. To access test site, the corresponding url would be  test.materialscommons.org/api .

For example, suppose the apikey for a given user is "12345678901234567890". Then that users might have a file  ~/.materialscommons/config.json which contained the following json:
{
    "apikey": "12345678901234567890",
    "mcurl": "http://mctest.localhost/api"
}

Calls to the API

Calls to the API are made through calls in Python code, either to functions in the Module or to methods of Objects returned by those calls. Each call returns either an object or a list of objects, and most calls have the side effect of changing the one or more objects in the database. When Object are changed in the database, an updated object is returned which in general is not the same as the object used in making the call. For objects representing the same underlying value in the database, the is especially important to note: after a call to update the object, in general the returned object is not equal the the object used in the call. They will have many of the same data values, but only the id values can be used to verify the identity of the object being represented. See the example below.


Many APIs return an object. This occurs when you request a specific object, such as a project, or you update or create one. Where the object in question has a list of dependent types, the API will return an overview of those type, rather than the details.

For example a Project contains a list of Samples. When requesting a Project you can ask that the full details for the Samples be included rather than the overview defintions. This is more expensive and should be used only when you will immediately need access to this information.

The details parameter allows you to define the level of details to return. By default the API returns details at overview level. The details parameter accepts overview or full as arguments. When full is specified you will receive a complete object definition for all dependent objects.

Examples

Projects

All access to projects requires an APIKEY to authenticate the request. The APIKEY controls access to the project and its resources.

Type definitions

Overview type

All objects have an overview definition. All overview definitions have the same fields. You can identify the type of overview object by looking at its _type field.

Overview type definition

type OverviewType struct {
    id          string      // The ID of the object
    name        string      // Name of the object
    description string
    ctime       integer     // Creation time of the object
    mtime       integer     // Last modification time
    _type       string      // Type of Object
}

Example data

{
    "id": "367416f9-c266-46cd-a224-f48f08c53b46",
    "name": "My Sample",
    "description": "Pure Aluminum from Ford",
    "ctime": 531360000,
    "mtime": 531360000,
    "_type": "sample"
}

Project type

type Project struct {
    id              string      // Unique ID of project
    name            string      // Name of project
    description     string      // Project description
    directory_id    string      // Top level directory id
    processes       []Process   // Processes in project
    samples         []Sample    // Samples in project
    users           []User      // Users with permissions to access project
    owner           string      // Owner of project
    ctime           time.Time   // Create time
    mtime           time.Time   // Last modification time
    _type           string      // "project"
}

Process type

type Process struct {
    id              string
    ctime           integer
    mtime           integer
    owner           string
    name            string
    description     string
    why             string
    owner           string
    directory_id    string
    mediatypes      MediaType
    samples_in      []Sample
    samples_out     []Sample
    files_in        []File
    files_out       []File
    measurements    []Measurement
    size            integer64
    _type           string
}

Sample type

type Sample struct {
    id              string
    property_set_id string
    name            string
    description     string
    files           []File
    processes       []Process
    measurements    []Measurement
}

User type

type User struct {
}

REST API

Authenticate

POST /login Authenticate the system and get back an authentication token. You may supply either a username and password or an application key, application id and username. Each application must be pre authorized by the user.

Parameters

Name Type Description
username string The users login id, for example username:user@umich.edu
password string The users account password
application_key string The system assigned application key
application_id string The application id associated with the application_key

Example using username and password

{
    "username": "user@umich.edu",
    "password": "my secure password 123"
}

Example using application parameters

{
    "username": "user@umich.edu",
    "application_key": "367416f9-c266-46cd-a224-f48f08c53bbb",
    "application_id": "mc_upload_client"
}

Create Project

POST /projects

The Create Project command will create a new project for the currently authenticated user. The project name must be unique for the user.

Input

name Type Description
name string The name of the project.
Name must be unique for the user.
description string Optional description for project.

Response Codes

Code Description
201 Created Project was successfully created
400 Bad Request The inputs were incorrect or additional
409 Conflict The user already has a project of the given name

Example

$ curl -X POST -H "Content-Type: application/json" \
         -d '{"name": "My Project", "description": "Test WE43 Properties"}' \
         https://materialscommons.org/api/v2/projects

Update Process in Project

PUT /projects/:project_id/processes/:process_id

The Update Process in Project updates attributes of the process. This command is structured as a series of commands. Each command works on different parts of the process. The process data structure is complex and updates to the structure can involve modifying, adding or deleting attributes associated with the process.

Command Description
samples Add or delete samples from the process
files Add or delete files from the process
measurements Add modify or delete measurements from the process
attributes Modify attributes such as name and description for the process
setup Add, modify or delete setup attributes of the process

Command Structure

{
    "samples": [],
    "files": [],
    "measurements": [],
    "attributes": [],
    "setup": []
}

samples command

The samples command updates a process by adding, modifying or removing samples. The command can modify a sample in the following ways:

  • Remove a file association from the sample. The file must have been associated with the sample in this process.
  • Add a file association to the sample. The file must already be associated with the process.
  • Delete a file association from a sample. The file must already be associated with the process.
Samples add command

    "command":"add",
    "sample_id": "the sample id"
}

    "sample_id": "the sample id"
}

Get Project Files

POST /projects/:project_id/files This command takes a list of file ids and returns the details for those files

It's parameter looks as follows:

{
    file_ids:[]
}

Where file_ids is a list of file ids belonging to the project.

Update Project File

PUT /projects/:project_id/files/:file_id

The Update Project File API updates attributes associated with the a file.

Command Description
name Change the name of the file
directory Change the directory the file is located in
notes Add, modify or delete a file note
tags Add or delete file tags
description Modify the file description

Command structure

{
    "name": "file name",
    "directory": "new directory id",
    "notes":[],
    "tags": [],
    "description": "the file description",
    "processes": [],
    "samples": []
}

notes command

The notes command takes a list of notes and updates the file to include the new set of notes. To delete a note you need to include the note with its note id and add the field "delete":true. This note entry will look as follows:

{
    "id": "the note id",
    "delete": true
}

To add a new note simply include a note without an id field. This will then be added as a new note to the file.

tags commands

The tags command takes a list of tags. The server then determines how that tags list has changed and updates the file with the new list of tags.
